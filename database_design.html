1. OLTP vs OLAP

Key Question: How should we organize and manage data?

The answer depends on how the data will be used. Different schemas, management options, and objects affect storage, access, query speeds, memory usage, and costs.

OLTP (Online Transaction Processing)

  • Purpose: Support day-to-day operations
  • Design: Application-oriented (e.g., bookkeeping)
  • Data: Up-to-date, operational (current snapshot)
  • Size: Gigabytes
  • Queries: Simple transactions, frequent updates
  • Users: Thousands (employees, customers)

Example OLTP Tasks:

  • Track and find book prices
  • Record customer transactions
  • Track employee hours

OLAP (Online Analytical Processing)

  • Purpose: Report and analyze data
  • Design: Subject-oriented (e.g., quarterly sales analysis)
  • Data: Consolidated, historical
  • Size: Terabytes (archive)
  • Queries: Complex aggregations, limited updates
  • Users: Hundreds (analysts, data scientists)

Example OLAP Tasks:

  • Analyze most profitable books
  • Find most loyal customers
  • Determine employee of the month

Relationship Between OLTP and OLAP

  • OLTP data is pulled and cleaned to create OLAP data warehouses
  • OLAP analysis informs business practices, influencing OLTP operations
  • They need each other!

💡 Key Insight: Before implementing anything, determine if you need OLTP, OLAP, or both based on business requirements.

2. Storing Data

Data Structure Levels

TypeDescriptionExamples
StructuredDefined by schemas, data types, tables, relationshipsSQL databases
UnstructuredSchemaless, raw formPhotos, chat logs, MP3s, raw text
Semi-structuredSelf-describing, some structureNoSQL, XML, JSON

Trade-off: Structured data is easier to analyze but less flexible/scalable. Unstructured data is more flexible but harder to analyze.

Storage Solutions

Traditional Databases

  • Follow relational schema
  • Used for OLTP with real-time structured data

Data Warehouses

  • Optimized for read-only analytics (OLAP)
  • Combine data from multiple sources
  • Use Massively Parallel Processing (MPP)
  • Use dimensional modeling and denormalized schemas
  • Examples: Amazon Redshift, Google BigQuery, Azure Data Warehouse

Data Mart

  • Subset of data warehouse
  • Dedicated to specific topic (Sales, HR, Marketing)
  • Easier departmental access

Data Lakes

  • Store all types of data at lower cost
  • Use object storage (cheaper than block/file storage)
  • Petabytes in size
  • Schema-on-read (schema created when data is read)
  • Run big data analytics with Apache Spark, Hadoop
  • Warning: Must be organized well or becomes a "data swamp"

3. Database Design Fundamentals

Core Concepts

Database Models: High-level specifications for database structure

  • Most popular: Relational model (rows = records, columns = attributes)
  • Others: NoSQL, object-oriented, network model

Schemas: Database blueprint implementing the model

  • Defines specific tables, fields, relationships, indexes, views
  • Must be respected when inserting data

Three Levels of Data Modeling

LevelDescriptionTools
ConceptualWhat database contains (entities, relationships, attributes)ER diagrams, UML
LogicalHow entities map to tablesDatabase models, schemas
PhysicalHow data is physically storedStorage specifications

4. Dimensional Modeling

Overview

  • Adaptation of relational model specifically for data warehouses
  • Optimized for OLAP queries (analyze, not update)
  • Uses star schema
  • Easy to interpret and extend

Elements

Fact Tables

  • Hold key metrics that change often
  • Contain foreign keys to dimension tables
  • Decided by business use-case

Dimension Tables

  • Hold descriptive attributes
  • Don't change as often
  • Expand on fact table attributes

SQL Example: Creating Dimension Tables

-- Create route dimension table
CREATE TABLE route (
    route_id INTEGER PRIMARY KEY,
    route_name VARCHAR(160) NOT NULL,
    city_name VARCHAR(160) NOT NULL,
    distance_km FLOAT NOT NULL,
    park_name VARCHAR(160) NOT NULL
);

-- Create week dimension table
CREATE TABLE week (
    week_id INTEGER PRIMARY KEY,
    week INTEGER NOT NULL,
    month VARCHAR(160) NOT NULL,
    year INTEGER NOT NULL
);

5. Star and Snowflake Schema

Star Schema

The simplest form of dimensional model.

Characteristics:

  • Fact table at center
  • Dimension tables extend outward (like star points)
  • One-to-many relationships (one store → many sales)

Snowflake Schema

Extension of star schema with normalized dimension tables.

Key Differences:

  • Same fact table as star schema
  • Dimension tables are further normalized (broken into more tables)
  • More tables, more joins

Adding Foreign Keys

-- Add foreign key constraints
ALTER TABLE fact_booksales ADD CONSTRAINT sales_book
    FOREIGN KEY (book_id) REFERENCES dim_book_star (book_id);
    
ALTER TABLE fact_booksales ADD CONSTRAINT sales_time
    FOREIGN KEY (time_id) REFERENCES dim_time_star (time_id);
    
ALTER TABLE fact_booksales ADD CONSTRAINT sales_store
    FOREIGN KEY (store_id) REFERENCES dim_store_star (store_id);

6. Normalization

Definition

Technique that divides tables into smaller tables connected via relationships.

Goal: Reduce redundancy and increase data integrity.

Normalized vs Denormalized

AspectNormalized (Snowflake)Denormalized (Star)
TablesMore tablesFewer tables
JoinsMore joins neededFewer joins
Query SpeedSlower readsFaster reads
StorageLess redundancyMore redundancy
UpdatesSafer, simplerRiskier
Best ForOLTP (write-intensive)OLAP (read-intensive)

Benefits of Normalization

  1. Saves storage space (eliminates redundancy)
  2. Ensures data integrity - Enforces consistency, referential integrity
  3. Safer modifications - Less records to alter, update once, reflected everywhere
  4. Easier to extend - Smaller, organized tables

7. Normal Forms (1NF, 2NF, 3NF)

Hierarchy (Least to Most Normalized)

1NF → 2NF → 3NF → BCNF → 4NF → 5NF → 6NF

First Normal Form (1NF)

Rules:

  • Each record must be unique (no duplicate rows)
  • Each cell must hold only one value

Second Normal Form (2NF)

Rules:

  • Must satisfy 1NF
  • If primary key is single column → automatically 2NF
  • If composite primary key → each non-key column must depend on ALL keys

Third Normal Form (3NF)

Rules:

  • Must satisfy 2NF
  • No transitive dependencies - non-key columns can't depend on other non-key columns

Data Anomalies (Why Normalize?)

AnomalyDescriptionExample
UpdateInconsistency from redundant data updatesChanging city name in some records but not others
InsertionCan't add records due to missing required dataCan't add student not enrolled in any course
DeletionUnintentionally deletes other dataDeleting last course record loses student info

💡 3NF prevents all three types of anomalies!

8. Database Views

What is a View?

  • Virtual table defined by a stored query
  • Not stored in memory (only the query is stored)
  • Data comes from actual tables
  • Query like a regular table

Creating Views

CREATE VIEW view_name AS
SELECT col1, col2
FROM table_name
WHERE condition;

-- Example
CREATE VIEW scifi_books AS
SELECT title, author
FROM books
WHERE genre = 'science fiction';

-- Query the view
SELECT * FROM scifi_books;

Benefits of Views

  1. No storage overhead (except for query statement)
  2. Access control - restrict what users can see
  3. Mask complexity - simplify normalized schema queries
  4. Reusability - no need to retype common queries

Managing Views

-- Grant privileges
GRANT UPDATE ON ratings TO PUBLIC;

-- Revoke privileges
REVOKE INSERT ON films FROM db_user;

-- Drop view
DROP VIEW view_name CASCADE;

-- Redefine view
CREATE OR REPLACE VIEW view_name AS new_query;

9. Materialized Views

Comparison

Non-Materialized ViewMaterialized View
VirtualPhysically stored
Stores queryStores query results
Runs query each timeQuery pre-computed
Always currentNeed to refresh

When to Use Materialized Views

  • Queries with long execution time
  • Underlying data doesn't change often
  • Data warehouses (OLAP - read-intensive)
  • Save on computational costs

Implementation in PostgreSQL

-- Create materialized view
CREATE MATERIALIZED VIEW my_view AS 
SELECT * FROM table;

-- Refresh when data changes
REFRESH MATERIALIZED VIEW my_view;

10. Database Roles and Access Control

What are Roles?

  • Manage database access permissions
  • Define privileges (login, create databases, write to tables)
  • Can be assigned to one or more users
  • Global across all databases in cluster

Creating Roles

-- Empty role
CREATE ROLE data_analyst;

-- Role with attributes
CREATE ROLE intern WITH PASSWORD 'PasswordIntern!' VALID UNTIL '2020-01-01';

-- Role that can create databases
CREATE ROLE admin CREATEDB;

-- Role with multiple abilities
CREATE ROLE admin WITH CREATEDB CREATEROLE;

Grant and Revoke Privileges

-- Grant privileges
GRANT UPDATE ON ratings TO data_analyst;
GRANT UPDATE, INSERT ON long_reviews TO data_scientist;

-- Revoke privileges
REVOKE UPDATE ON ratings FROM data_analyst;
REVOKE UPDATE, INSERT ON long_reviews FROM PUBLIC;

11. Table Partitioning

Why Partition?

  • Large tables (100GB+) become slow to query
  • Indexes don't fit in memory
  • Solution: Split table into smaller parts

Types of Partitioning

Vertical Partitioning

Split by columns

  • Create separate table for rarely-queried columns
  • Link through shared key (id)
  • Can store cold data on slower medium

Horizontal Partitioning

Split by rows

  • Commonly by timestamp or category
  • Each partition holds subset of rows

PostgreSQL Horizontal Partitioning

CREATE TABLE sales (
    ...
    timestamp DATE NOT NULL
)
PARTITION BY RANGE (timestamp);

CREATE TABLE sales_2019_q1 PARTITION OF sales
    FOR VALUES FROM ('2019-01-01') TO ('2019-03-31');

CREATE TABLE sales_2019_q4 PARTITION OF sales
    FOR VALUES FROM ('2019-10-01') TO ('2019-12-31');

12. Data Integration

Definition

Combining data from various sources, formats, and technologies into a unified view.

Use Cases

  • 360° customer view: All departments' data in one place
  • Acquisitions: Combining company databases
  • Legacy systems: Query all historical data at once

Best Practices

  1. Automated testing & alerts: Detect data corruption
  2. Security: Maintain original access restrictions
  3. Data governance/lineage: Know data origin and usage

13. Choosing a DBMS

SQL DBMS (RDBMS)

Based on relational model

Examples: SQL Server, MySQL, PostgreSQL, Oracle SQL

Best when:

  • Structured, unchanging data
  • Predefined schema benefits you
  • Data consistency is critical (e.g., accounting)

NoSQL DBMS

Non-relational, less structured

Best when:

  • Rapid growth, unclear schema
  • Large data quantities
  • Varying data structures
